# M3 驗收測試手冊

> 用途：M3「監聽 + 觸發 + Logger」各項功能的人工驗收清單。自用情境隨時抽選測試，不必一次跑完。
>
> 設計：每項都提供「**最簡驗證**」（信任設計、看 state）+「**嚴謹驗證**」（走真實 `_evaluate` 路徑）兩條路。實務上看 contextState 對就 90% 確定設計通了。

---

## 0. 預備

### 啟動
```powershell
npm run start:dev
```
DevTools 視窗會自動 detached 開出（不在桌寵那個透明視窗裡）。

### Console 解鎖貼上
DevTools → Console tab，第一次貼上前在 prompt 手動打字：
```
allow pasting
```
按 Enter。當前 session 有效；關掉 DevTools 重開要再做一次。

### 主要工具

| 工具 | 用途 |
|---|---|
| `dev-box` 5 顆 M3 按鈕 | 計數 / 狀態 / 觸發 / 重置CD / 清空 |
| DevTools console | `await window.api.debug.*()` 系列 |
| `node scripts/dump-events.js` | events JSONL 轉譯人類可讀 |
| `文件/events-schema.md` | 各 event type 欄位速查 |

### 9 個 debug IPC 速查

```js
await window.api.debug.countersGet()        // InputMonitor.snapshot()
await window.api.debug.envGet()             // 全 plugin snapshot + active capabilities
await window.api.debug.contextStateGet()    // 7 個高階 state（含 confidence/sources/ttl）
await window.api.debug.pluginsStatus()      // 每個 plugin 的 enabled/healthy/lastError
await window.api.debug.rulesStatus()        // 規則啟用狀態 + 缺哪個 capability
await window.api.debug.triggerHistory(50)   // 最近 N 筆 trigger:fired
await window.api.debug.flushEvents()        // 強制 flush buffer
window.api.debug.fire('click_too_much')     // forceFire（繞 cooldown 跟 suppress，dev 用）
window.api.debug.resetCooldowns()           // 清空 cooldown
```

## 2. 待測項目（按優先排序）

### ⭐⭐⭐ 2.1 `meeting_silence` — 通話中靜音

**已半驗證**（context state 正確 0.7）。剩下嚴謹版：

#### 最簡驗證（已過）
1. 開 Discord/Teams/Zoom 語音通話
2. DevTools console:
   ```js
   await window.api.debug.contextStateGet()
   ```
3. 預期 `in_meeting`:
   ```js
   { value: true, confidence: 0.7+, sources: ["mic_active", "audio_mic_combo"], ... }
   ```

#### 嚴謹驗證（未做，需 5-10 min）
1. 通話中 → dev-box 點「重置CD」
2. 在桌面**連點 500 下滑鼠**（走真實 `_evaluate()` 路徑，不要按 dev-box「觸發」按鈕，那是 forceFire 繞過 suppress）
3. 桌寵應該**完全沒冒對話**
4. 確認：
   ```powershell
   node scripts/dump-events.js --since 5m --types trigger:fired
   ```
   通話期間應該**沒有 click_too_much**（drag 仍會 fire 因為 priority 100 是壓制例外）
5. 掛斷通話 60 秒後（mic-cam-activity 5 秒輪詢 + ContextState TTL 60 秒）→ 連點 500 下 → 又能 fire

#### 注意
- `forceFire` (dev-box「觸發」按鈕) **繞過 suppress 檢查**，故意設計給測 UI 用，不能驗證壓制。要驗實際壓制只能走真實 counter。
- 「**純聽會議不開麥**」**不會觸發 in_meeting**（mic_active=false）— 設計如此（沒人聽得到你發話，桌寵冒泡無妨）

### ⭐⭐ 2.2 `game_silence` — 全螢幕遊戲靜音

#### 最簡驗證
1. 開全螢幕遊戲（exe 在 [config/app-classification.json](../config/app-classification.json) 的 `games` 清單）
2. 玩 30 秒讓 GPU 持續高
3. DevTools console:
   ```js
   await window.api.debug.contextStateGet()
   ```
4. 預期 `in_game`:
   ```js
   { value: true, confidence: ≥0.6, sources: ["fullscreen", "gpu_high", "game_exe"] }
   ```

#### 反向測試 — 不該誤判
- 開 Blender/Premiere/OBS 全螢幕（不在 games 清單）
- 預期 `in_game.confidence` 落在 0.4-0.5（fullscreen + gpu_high 但缺 game_exe）→ `value: false`
- 多源融合的設計目的就在這

#### 我的遊戲 exe 不在清單
- 編輯 `config/app-classification.json`
- 加進 `games` 陣列（exe 名小寫，含 `.exe`）
- 重啟程式生效

#### 嚴謹驗證
連點 500 下 → 沒 trigger:fired（除 drag）

### ⭐⭐ 2.3 `ide_calm_down` — IDE 工作降頻率

設計詳解見 [§ in_ide 機制](#附錄-in_ide-降頻率機制) 在附錄。

#### 最簡驗證（30 秒）
1. 切 VSCode（或任何 IDE）為前景
2. DevTools console:
   ```js
   await window.api.debug.contextStateGet()
   ```
3. 預期 `in_ide: { value: true, confidence: 0.9, sources: ["ide_exe"] }`
4. 切回 Chrome → 應該 `value: false`

切換有反應就 OK。實際 multiply_cooldown 邏輯是程式碼層級，不必驗。

#### 嚴謹驗證（22 分鐘，不必做）
1. 切 VSCode 前景 → dev-box 點「重置CD」
2. 連點 500 下 → 第一次 click_too_much fire
3. 等 11 分鐘 → 連點 500 下 → 應被擋（cooldown 變 1200 秒）
4. 等到累積 21 分鐘 → 連點 500 下 → 能 fire
5. 切回 Chrome 重複 → cooldown 600 秒生效

#### 我的 IDE 不在清單
編輯 `config/app-classification.json` 的 `ides` 陣列，加進你的編輯器 exe。

#### 想改強度
編輯 `config/triggers.json` 的 `ide_calm_down`：
```jsonc
{ "factor": 3.0 }   // 預設 2.0；改 3.0 變 30 分鐘 cooldown
```
或直接改 `"action": "suppress_all"` 全閉嘴。

### ⭐⭐⭐ 2.6 Plugin level 切換 — 「外掛可選擇運作到哪個階段」

這是你提案的核心特性，**值得花 5 分鐘驗證**。

#### 步驟
1. PowerShell `Ctrl+C` 結束目前 dev session
2. 編輯 `config/plugins.json`：
   ```jsonc
   "monitor_level": "tier1"
   ```
3. `npm run start:dev` 重啟
4. DevTools console:
   ```js
   await window.api.debug.pluginsStatus()
   ```
5. 預期所有 tier2/tier3 plugin 出現 `failureReason: 'disabled-by-config', enabled: false` 
6. 跑：
   ```js
   await window.api.debug.contextStateGet()
   ```
7. 預期 **8 個 state 全部** `value: null, reason: 'capability-missing'`（包含 `focused_work` — 它需要 `keyboard_input` + `foreground_window` 雙 capability，tier1 無 window-tracker 也會缺）
8. 等 1 分鐘後跑：
   ```powershell
   node scripts/dump-events.js --since 1m --summary
   ```
9. 類型分布只有 `typing-burst / mouse-burst / click-burst / click / idle-*`（沒 power/window/system/audio）

#### 切回 tier1+2
```jsonc
"monitor_level": "tier1+2"
```
重啟 → 加上 power/screen/theme/window/fullscreen 事件，仍無 tier3。

#### 個別 override 測試
```jsonc
"monitor_level": "tier1+2+3",
"plugins": {
  "tier3-clipboard-watcher": { "enabled": false }
}
```
重啟 → 全部都在但 `clipboard:changed` 事件不再出現。

#### 切回完整模式
```jsonc
"monitor_level": "tier1+2+3"
```
（移除個別 override）

## 3. 24 小時穩定性（被動跑）

**不必特別保留時段**，日常使用順便驗。隔天起床跑：

```powershell
node scripts/dump-events.js --summary
```

### 健康指標
| 指標 | 預期 |
|---|---|
| events 檔大小 | < 50 MB（外推 24h 約 10-30 MB）|
| 類型分布 | 鍵滑佔大宗、tier2/tier3 事件零星 |
| `plugin:degraded` 計數 | **0 筆**（任何非 0 都要查）|
| 工作管理員 electron 主程序 RSS | < 500 MB |

### 中途某 plugin 掛了？
```powershell
node scripts/dump-events.js --types plugin:degraded
```
看 `plugin` / `reason` / `error` 三個欄位定位問題。

## 附錄

### in_ide 降頻率機制

`ide_calm_down` 規則 priority 997，action 是 `multiply_cooldown` (factor 2.0)。觸發時把所有規則的 effective cooldown 變兩倍：

| 規則 | base | × ide(2.0) | = effective |
|---|---|---|---|
| click_too_much | 600s | × 2 | 20 分 |
| long_idle | 1800s | × 2 | 60 分 |
| continuous_use | 1200s | × 2 | 40 分 |
| deep_night | 3600s | × 2 | 120 分 |
| drag_character | 5s | × 2 | 10 秒（priority 100 仍可觸發）|

#### 跟動態冷卻相乘

`final_cooldown = base × ide_factor × dynamic_factor`

dynamic 看最近 60 分鐘 trigger 數：
- ≥8 → ×0.7（高活躍縮短）
- ≤2 → ×1.5（低活躍拉長）
- 中間 → ×1.0

寫 code 中 + 桌寵嘴你很多次：`600 × 2 × 0.7 = 14 分`
寫 code 中 + 桌寵很少嘴：`600 × 2 × 1.5 = 30 分`

#### 為什麼是降頻率不是全靜音

寫 code 久了**仍需要**被提醒休息（避免過勞）。`suppress_all` 完全閉嘴會錯過該被打擾的時刻。`multiply_cooldown` 是「降低存在感」不是「消失」。

如果你想完全閉嘴，把 `triggers.json` 的 `ide_calm_down.action` 改成 `"suppress_all"` 即可。

### 規則 priority 對照表

| priority | 規則 | 行為 |
|---|---|---|
| 999 | meeting_silence / game_silence / screen_locked_silence | suppress_all（除 priority ≥ 100）|
| 998 | video_partial_silence | suppress_categories（部分壓制）|
| 997 | ide_calm_down | multiply_cooldown ×2（降頻率）|
| 100 | drag_character | 例外：壓制不適用 |
| 60 | deep_night | 時段觸發 |
| 30 | long_idle | 閒置觸發 |
| 25 | continuous_use | 連續使用觸發 |
| 10 | click_too_much | 計數觸發 |

### Cooldown 不持久化

`TriggerEngine._lastFireByCategory` 是記憶體 Map，**程式重啟即清空**。重啟後第一次評估若條件滿足，會立刻 fire（不會等舊 cooldown）。這是**設計選擇**：

- 開機後桌寵應有第一次互動，不該因為「昨天剛 fire 過 deep_night」就靜默 1 小時
- 副作用：頻繁開發重啟時規則會異常多 fire
- 可改進方案（M4+ 視需求）：持久化到 `data/cooldowns.json`、或重啟後隨機延遲首次評估

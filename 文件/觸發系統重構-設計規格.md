# 觸發系統重構設計規格 v1.1

> ✅ 使用者已對 §12 5 個 Q + 既有內容重構（R1-R5）拍板，本文件**已定案**。

## 文件目的

把原本 5+1 條觸發規則 + 6 個 category 系統，重構成**多元觸發 + 分軌語音 + 互動式氣泡**的架構，讓桌寵在正常使用時段有充分互動感、提供主動互動入口、語音與時間日期分離以解決動態時間表達。

---

## §1 現況盤點與痛點

### 既有觸發

| 規則 | 條件 | 一天頻率 |
|---|---|---|
| `drag_character` | 拖角色 event | 看你拖（測試常觸發）|
| `deep_night` | 23:00-05:00 | ~1/小時 |
| `long_idle` | idle ≥ 30 分 | 3-5/天 |
| `continuous_use` | session ≥ 4 小時 | 0-3/天 |
| `click_too_much` | 累計 ≥ 500 click | 不固定 |
| `hourly_chime` | minute=0, hour 8-22 | 15/天（剛加，未自然觸發）|
| **5 條 suppressor** | meeting/game/locked/video/IDE | 抑制 |

**既有 categories**: click_too_much / long_idle / continuous_use / deep_night / drag (各 33 句) + hourly_chime (2 句)

### 使用觀察（從你 events log）

- 8:00-23:00 普通使用：**平均 30-45 分鐘才 1 次觸發**（太冷清）
- 觸發集中在 18-20 點開發高峰（71% 今天觸發都在這裡）
- 87% 觸發走 haiyin（活躍 persona）

### 痛點

1. ❌ **正常使用時段冷清**：1-4 小時 session 內幾乎只能等 hourly_chime
2. ❌ **沒主動互動入口**：用戶想跟角色互動只能拖（不直觀）
3. ❌ **時間語音失效**：dialogues 含 `{time}` `{hour}` 變數，但語音是預生的，TTS 念不出當下時間
4. ❌ **氣泡類型 5×4×3 種沒文案**：speech-bubble 支援 thought/narration/system/whisper + choice/binary 互動，**但 dialogues 全是 speech + advance**，浪費既有 UI 能力
5. ❌ **無對話樹**：點擊角色只能單句回應，無「選項分支」對話

## §2 設計目標

| 目標 | 衡量 |
|---|---|
| G1 提升正常使用時段互動感 | 每 5-15 分鐘有對話冒出 |
| G2 主動互動入口 | 點擊立繪能觸發 + choice 選項回應 |
| G3 時間語音正確 | 「14:00」這類時間動態正確念出 |
| G4 充分用既有氣泡類型 | thought / whisper / choice / binary 各有對應文案 + 觸發 |
| G5 不增加噪音感 | 動態 cooldown / suppressor / 機率 gate 控制 |
| G6 易擴充 | 加新觸發類型不用改 UI / schema |

## §3 整體架構

### 觸發 → 對話 → 語音 完整 pipeline

```
[輸入事件]                  [評估]                       [選句]                  [呈現]
─────────                  ─────                       ─────                  ─────
input/system/user      →  TriggerEngine            →  DialogueDirector  →   renderer
                          ├ 事件規則 (event)            ├ 拿 sequence       ├ 文字氣泡
                          ├ 計數規則 (counter)          ├ 看 voice_prefix   │  (依 type/interaction
                          ├ 時間規則 (time_marker)      ├ 串接時間音 + 情境音│  渲染 choice 等)
                          ├ 狀態邊緣 (state_edge)       ├ recent ring 過濾   ├ 語音 sequential 播放
                          ├ 隨機間隔 (random_interval)  └ 推 dialogue:show   │  [time.wav, voice.wav]
                          └ App 切換 (app_focus)        └ 推 voice:play       └ 選項處理
                                                                              └ 串接互動下一輪
```

### 分軌語音架構（核心改動）

```
觸發 hourly_active (14:00)
   ↓
voice 串接：
   1. personas/<id>/voices-time/zh/14_dian.wav     ← 預生「14 點」
   2. personas/<id>/voices/zh/chatter_007_0.wav    ← 預生情境句
   ↓
renderer 順序播：先播時間音（~1 秒）→ padding 150ms → 播情境音
   ↓
氣泡同時顯示：「14 點了～累了嗎？」（合成顯示，但語音是 2 段）
```

**關鍵概念**：時間部分從 dialogue 句子裡抽出去，做成獨立預生音庫。情境句永遠不含時間變數。

## §4 觸發類別詳細設計（三軸）

### §4.1 軸 1：事件滿足觸發（5 條新規則）

#### 4.1.1 typing_burst_milestone — 短時間頻繁打字
```jsonc
{
  "name": "typing_burst_milestone",
  "category": "typing_burst",
  "priority": 25,
  "cooldown_sec": 300,
  "voice_prefix": null,
  "condition": {
    "type": "event_burst",       // 🆕 新 condition type
    "event": "typing-burst",
    "window_sec": 60,
    "min_count": 3
  }
}
```
**語意**：「最近 1 分鐘有 3 次打字爆發 → 拼了一段 → 角色關心一下」

#### 4.1.2 click_burst — 短時間頻繁點擊（區別 click_too_much）
```jsonc
{
  "name": "click_burst",
  "category": "click_burst",
  "priority": 20,
  "cooldown_sec": 180,
  "condition": {
    "type": "event_burst",
    "event": "click",
    "window_sec": 30,
    "min_count": 50
  }
}
```
**語意**：「30 秒 50 點 → 在忙什麼急的事 → 提醒一下」（vs click_too_much 的累計 500 點，是長期過量）

#### 4.1.3 app_focus_xxx — 切到常用 app
```jsonc
{
  "name": "app_focus_browser",
  "category": "app_switch",
  "priority": 15,
  "cooldown_sec": 1800,
  "condition": {
    "type": "app_focus",         // 🆕 新 condition type
    "classifications": ["browser"],
    "probability": 0.3
  }
}
// 同款另開 chat / video editor / etc
```
**語意**：「切到瀏覽器，30% 機率冒一句」

#### 4.1.4 game_enter — 進入遊戲（state edge）
```jsonc
{
  "name": "game_enter",
  "category": "game_enter",
  "priority": 80,
  "cooldown_sec": 3600,
  "condition": {
    "type": "state_edge",        // 🆕 新 condition type
    "state": "in_game",
    "from": false,
    "to": true,
    "min_confidence": 0.6
  }
}
```
**語意**：「`in_game` 從 false 變 true → 玩家剛進遊戲 → 冒一句『要打遊戲了？』」**邊緣觸發只一次**，後續 game_silence suppressor 全擋。

#### 4.1.5 boot_greet — 開機問候
```jsonc
{
  "name": "boot_greet",
  "category": "boot_greet",
  "priority": 90,
  "cooldown_sec": 0,             // 啟動才 reset，重啟前不會 fire 第二次
  "condition": {
    "type": "event",
    "event": "app:ready"         // 🆕 main.js 啟動時推
  }
}
```

### §4.2 軸 2：時間觸發（5-6 條，**含時間音串接**）

#### 4.2.1 hourly_active — 整點報時（持續使用時）⭐
```jsonc
{
  "name": "hourly_active",
  "category": "chatter",         // ⚠ 共用 chatter category
  "priority": 50,
  "cooldown_sec": 3540,
  "voice_prefix": "hour_dynamic",   // 🆕 動態：當下時間決定哪個時間音
  "condition": {
    "type": "time_marker",
    "minute": 0,
    "hour_range": [8, 22],
    "min_active_sec": 60         // 🆕 過去 60 秒有 input 才觸發
  }
}
```
**語意**：整點 + 你還在用 → 串接「14 點」+ chatter 句。

#### 4.2.2 random_chatter — 隨機閒談 ⭐
```jsonc
{
  "name": "random_chatter",
  "category": "chatter",
  "priority": 40,
  "cooldown_sec": 600,
  "voice_prefix": null,            // 不串時間音
  "condition": {
    "type": "random_interval",     // 🆕 新 condition type
    "probability_per_eval": 0.005,
    "hour_range": [8, 22]
  }
}
```
**語意**：cooldown 後每秒 0.5% 機率 → 平均 ~13 分鐘冒一次。

#### 4.2.3-6 時段問候 4 條
```jsonc
{
  "name": "morning_greet",
  "category": "time_greet",
  "priority": 70,
  "cooldown_sec": 86000,           // ~1 天，避免一天兩次
  "voice_prefix": "morning",
  "condition": {
    "type": "time_marker",
    "minute": 0,
    "hour_range": [7, 7],          // 7:00 一次
    "min_active_sec": 60
  }
}
```
類似 4 條：morning_greet (7:00) / lunch_greet (12:00) / evening_greet (18:00) / night_greet (22:00)。

每條 voice_prefix 對應時段詞音檔：`morning.wav` / `lunch.wav` / `evening.wav` / `night.wav`。

### §4.3 軸 3：主動觸發（2 條）

#### 4.3.1 click_character — 點擊角色 ⭐
```jsonc
{
  "name": "click_character",
  "category": "click_character",
  "priority": 100,
  "cooldown_sec": 3,
  "condition": {
    "type": "event",
    "event": "character:click"     // 🆕 renderer 推
  }
}
```
**語意**：左鍵點立繪 → 觸發互動式 sequence（含 choice 選項）。

#### 4.3.2 click_streak — 連續點擊升級互動
```jsonc
{
  "name": "click_streak",
  "category": "click_character",
  "priority": 105,                  // 比 4.3.1 高，搶先 fire
  "cooldown_sec": 0,
  "condition": {
    "type": "streak_threshold",     // 🆕 新 condition type
    "event": "character:click",
    "window_sec": 10,
    "min_count": 5                  // 10 秒內連點 5 次
  }
}
```
**語意**：連點 5 下 → 「夠了喔 / 再點生氣」這種升級回應。

✅ **已決定（Q5: A）**：DialogueDirector 抽 sequence 時，依當下 streak 數從**單一** `click_character` category 內挑對應 `streak_level` 的句。schema 較乾淨，不額外切 category。

streak 對應 level：
| streak 數 | streak_level | 預期語感 |
|---|---|---|
| 1 | `low` | 普通互動、含 choice 選項 |
| 2-4 | `mid` | 微嗔、開始注意 |
| 5+ | `high` | 「夠了喔 / 再點要生氣了」|

`streak_threshold` rule 觸發時 director 強制只挑 `streak_level=high`；single click 觸發只挑 `low` / `mid`（按 streak 階梯）。

### §4.4 既有 165 句重構計畫（R1-R5） ✅ 已決定

使用者已拍板：**R1-R5 全重構**。既有 5 個 category（click_too_much / long_idle / continuous_use / deep_night / drag）共 165 句，全部過一遍重構。

#### R1. 移除時間 / 日期變數
**範圍**：包含 `{time}` `{hour}` `{date}` 等動態變數的句（粗估 ~30 條，集中於 deep_night / continuous_use）。
**處理**：

- 必須報時的 → 移到 chatter category（由 hourly_active 串時間音）
- 移除後 [src/main/dialogue-director.js](../src/main/dialogue-director.js) 的 `_interpolate` 邏輯仍保留（給 {persona_name} 之類用），但時間相關鍵將無對應句使用。

#### R2. 視覺類型多樣化
依 §6.2 配對表，把部分 sequence 從預設 `type=speech` 改成 `whisper` / `thought`：
- `deep_night`：60% 改 whisper（深夜低語自然），20% 改 thought（內心戲），20% 保 speech
- `continuous_use`：第 4+ 小時段落改 whisper（疲勞低語）
- `click_too_much`：部分改 thought（內心吐槽）
- `long_idle`：保 speech 為主（歡迎回來語感）
- `drag`：保 speech（直接互動）

#### R3. 加互動式 sequence
從既有句中挑出 **3-5 條 / category** 改成 choice 或 binary：
- `deep_night` +2 條 binary（「還不睡？」yes/no）
- `continuous_use` +2 條 binary（「休息一下吧？」yes/no）
- `long_idle` +1 條 choice（「想做什麼？」）
- `click_too_much` +1 條 binary（「需要幫忙？」）
- `drag` 不加（拖曳本身就是互動）

#### R4. 重新分類
部分句子原本放錯 category 或內容偏離主題的，移動到合適 category：
- 部分 deep_night 中**非時段性**句 → 移 chatter
- 部分 long_idle 中**通用閒談** → 移 chatter
- 部分 continuous_use 中**非疲勞語感** → 移 chatter

#### R5. 重生語音
所有改動的 sequence（包含 R1 文字改寫、R2 type 改變、R3 加 response、R4 移動）都需重新批次生語音：
- voice manifest hash 變動會自動 mark stale → batch-runner 重生
- 預估重生量：~80 條 sequence wav + ~10 條 response wav（合計 ~90 條，~10 分鐘 batch）

#### R1-R5 工序
- R1+R4 純文字操作 → 直接在 `dialogues-manager-window` 內編輯（人工，~1 天）
- R2 type 改動 → 同上 UI 編輯（~0.3 天）
- R3 加互動 → 寫 choice/binary schema + UI 支援編輯（~0.5 天，UI 工日已含在 P1 互動 schema）
- R5 重生語音 → 一鍵 batch（~10 分鐘等待）

**合計人工 ~2 天**（已計入 §9 估算）。

## §5 時間語音串接架構（核心新增）

### §5.1 時間音庫設計

確認：**整點 24 個 + 時段詞 4-6 個 ≈ 30 個內**（per persona）。

```
personas/<id>/voices-time/zh/
├── manifest.json                  # 時間音庫的索引
├── hour_00.wav                    # 「0 點」/「凌晨 0 點」
├── hour_01.wav
├── ...
├── hour_23.wav                    # 「23 點」/「晚上 11 點」
├── morning.wav                    # 「早安」
├── lunch.wav                      # 「中午了」
├── evening.wav                    # 「晚上好」
├── night.wav                      # 「晚安」
└── (optional) midnight.wav        # 「半夜了」
```

**ref audio**：跟 `voices/` 用同一份 ref（同 persona 同音色）。

**生成腳本**：Tab 5 加按鈕「生時間音庫」，跑 ~30 個音檔（< 2 分鐘）。

### §5.2 voice_prefix 規格

每條 trigger rule 可選加 `voice_prefix`：

| 值 | 行為 |
|---|---|
| `null` / 不寫 | 不串時間音，純情境音 |
| `"morning"` 等固定詞 | 串 `voices-time/morning.wav` + 情境音 |
| `"hour_dynamic"` | 動態取當下小時 → `voices-time/hour_HH.wav` + 情境音 |
| `"hour_period_dynamic"` | 動態判斷時段（早午晚夜）→ 對應時段詞音 + 情境音 |

### §5.3 串接播放時序

```
[t=0]   播 voices-time/hour_14.wav      持續 ~1.0 秒
[t=1.0] 靜音 padding                    150 ms
[t=1.15] 播 voices/chatter_007_0.wav    持續 ~2.5 秒
[t=3.65] 結束
```

renderer 端用 `audio.addEventListener('ended', playNext)` 串接，padding 用 `setTimeout`。

### §5.4 IPC schema 變動

`voice:play` payload 從 `{ file_path }` 改為 `{ file_paths: [...] }`：

```jsonc
// 既有（保留向後兼容）
{ "file_path": "abc.wav" }

// 新（支援串接）
{ "file_paths": ["voices-time/hour_14.wav", "voices/chatter_007_0.wav"] }
```

DialogueDirector handleFire 時依 rule.voice_prefix 組路徑陣列。

### §5.5 voices-time/ manifest 結構

```jsonc
{
  "schema": "v1",
  "ref_audio": "voice-refs/haiyin-ref.mp3",
  "ref_text": "...",
  "entries": {
    "hour_14": {
      "file": "hour_14.wav",
      "text": "下午兩點",          // 生成時用的文字
      "category": "hour",
      "generated_at": "2026-05-10T..."
    },
    "morning": { "file": "morning.wav", "text": "早安", "category": "period", ... }
  }
}
```

## §6 氣泡類型 × 觸發配對（新章節）

### §6.1 既有氣泡能力

| 維度 | 值 | 用途 |
|---|---|---|
| **type**（視覺）| speech | 對白氣泡（既有 99% 用這）|
|  | thought | 內心獨白（括號包文字、淡灰底，少量已用）|
|  | narration | 旁白 / 場景描述（**從未用**）|
|  | system | 系統訊息（**從未用**）|
|  | whisper | 低聲呢喃（**從未用**，深夜場景）|
| **interaction** | display | 純顯示自動消失（thought 預設）|
|  | advance | 點下一行（多 lines sequence 用）|
|  | choice | 多選互動（**從未用**）|
|  | binary | 二元互動（**從未用**）|
| **persistence** | transient | 自動消失（預設）|
|  | persistent | 手動 ✕ 關 |
|  | sticky | 半持久（M3 後）|
|  | pinned | 釘選最頂 |

### §6.2 各觸發推薦氣泡類型

| 觸發 / category | type | interaction | persistence | 理由 |
|---|---|---|---|---|
| **click_character**（連續）| speech | **choice** | transient | 點下去給選項，最自然的 RPG 感 |
| **click_character**（單次）| speech | display | transient | 簡單回應 |
| **chatter** | speech | display | transient | 純閒談、自動消失 |
| **time_greet** | speech | display | transient | 純問候、不期待回應 |
| **deep_night**（已有）| **whisper** | display | transient | 深夜用低語符合語境 |
| **morning_greet** | speech | **binary** | transient | 「早安」/ 「還在睡」二選一 |
| **lunch_greet** | speech | **binary** | transient | 「吃了？」/「沒」 |
| **game_enter** | speech | **binary** | persistent | 「打 LoL 嗎？」釘住等回 |
| **app_focus_browser** | speech | display | transient | 隨意一句 |
| **app_focus_chat** | **thought** | display | transient | 內心戲：「跟誰聊？」（暗黑感）|
| **boot_greet** | speech | display | persistent | 開機問候釘住 5 秒 |
| **typing_burst** | thought | display | transient | 「忙打字呢⋯⋯」內心 |
| **click_burst** | speech | display | transient | 「點得急」直接吐槽 |
| **continuous_use**（既有）| **whisper** + speech 混合 | display | transient | 第 4 小時起，提醒語氣可漸變 |
| **long_idle**（既有）| speech | display | transient | 歡迎回來語感 |
| **click_too_much**（既有）| speech / thought | display | transient | 維持既有 |
| **drag**（既有）| speech | display | transient | 維持既有 |

### §6.3 互動式對話樹設計（choice / binary）

#### 6.3.1 schema 擴充

既有 sequence schema：
```jsonc
{
  "sequenceId": "haiyin_click_001",
  "type": "speech",
  "interaction": "advance",  // 或 display
  "lines": [{ "text": "...", "expression": "happy" }],
  "_meta": {...}
}
```

新增 choice / binary 結構：
```jsonc
{
  "sequenceId": "haiyin_click_004",
  "type": "speech",
  "interaction": "choice",
  "lines": [{ "text": "想做什麼？", "expression": "happy" }],
  "choices": [
    {
```

#### 6.3.2 互動觸發流程

```
觸發 click_character (連續模式)
  ↓
DialogueDirector 抽 choice 類 sequence
  ↓
推 dialogue:show 給 renderer (含 choices 陣列)
  ↓
renderer SpeechBubble 渲染：
  「想做什麼？」
  [想抱抱] [陪我聊天] [沒事]
  ↓
使用者點某選項 → renderer 推 dialogue:choice-selected IPC
  ↓
main.js 接事件 → DialogueDirector 從 sequence.choices[i].response 推下一輪 dialogue:show
  ↓
renderer 顯示新句（含 expression / 語音）
  ↓
（單輪對話結束；下次觸發 click_character 重新進入）
```

#### 6.3.3 voice 對應 ✅ 已決定（Q3: A 全預生）

每個 choice / binary 的 response.text 都**必生對應 wav**，提供一致語音體驗。批次生語音時要遍歷 `sequence.lines` + `sequence.choices[].response` + `sequence.binary.{yes,no}.response`。

### §6.4 對話內容創作分類框架（9 類）

R1-R5 重構與新 category 內容創作的統一指引。每條 sequence 在創作時應對應到下面其中一類，以利 LLM prompt 指定數量配比與表達風格。

#### 9 類分類矩陣

| # | 情境類型 | 視角 | 期待回應 | 推薦 type | 推薦 interaction | 對應觸發例 |
|---|---|---|---|---|---|---|
| 1 | **思考 / 模擬思考過程** | 內心（第一人稱）| 無 | `thought` | `display` | typing_burst、click_too_much 部分 |
| 2 | **自言自語 / 評論** | 第一人稱 | 無 | `speech` / `whisper` | `display` | drag、long_idle、deep_night |
| 3 | **主動互動 / 閒聊** | 第一人稱 | 期待開放回應 | `speech` | **`choice`** / `advance` | click_character、chatter |
| 4 | **回應使用者** | 第一人稱 | （本身即回應）| `speech` | `display` | choice/binary 內 response |
| **A** | **指令 / 提醒** | 第一人稱 | 期待**行動** | `speech` | `binary` / `advance` | continuous_use、time_greet 午晚 |
| **B** | **詢問** | 第一人稱 | 期待封閉答案 | `speech` | **`binary` / `choice`** | lunch_greet、game_enter |
| **C** | **撒嬌 / 求關注** | 第一人稱 | 暗中期待 | `speech` / `whisper` | `display` / `advance` | long_idle 中段、click_too_much |
| **D** | **情緒反應** | 第一人稱 | 無 | `speech` | `display` | typing_burst、click_burst、app_switch |
| **E** | **情境旁白** | 第三人稱 | 無 | **`narration`** | `display` | 任何 trigger 少量穿插 |

#### 各類辨識準則（避免混淆）

- **1 思考 vs 2 自言自語**：思考是內心戲（不該被聽見），自言自語是說出口的（用 whisper 表低聲）。對應 `thought` vs `speech/whisper`。
- **2 自言自語 vs A 指令/提醒**：評論只是表達看法（無預期使用者動作），提醒則暗示「希望你照做」。文案上前者用陳述句、後者多用疑問或祈使。
- **3 主動互動 vs B 詢問**：閒聊是**開放式**邀請對談（multiple choice 或自由），詢問是**封閉式**問題（yes/no 或固定選項）。schema 上前者多用 `choice`，後者多用 `binary`。
- **3 主動互動 vs C 撒嬌**：閒聊是真的想聊（有 choices 走向），撒嬌只想被注意（不一定有 choices，多為一句帶情緒的 display）。
- **D 情緒反應 vs 2 評論**：情緒反應是即時感嘆（單字 / 短句、無分析），評論有完整陳述。例：「哇！」是 D，「這個畫面好棒呢」是 2。
- **E 情境旁白 vs 1 思考**：旁白是**第三人稱**描述場景（用 narration 氣泡），思考是**第一人稱**內心（用 thought 氣泡）。

#### LLM prompt 指定範例

寫 click_character 共 20 條時可指定：
```
類別 3（主動互動 / 閒聊）：8 條 — 含 5 條 choice 互動，3 條 advance
類別 C（撒嬌 / 求關注）：5 條 — display
類別 1（思考）：3 條 — thought display
類別 4（回應）：4 條 — 對應 choice/binary response（不獨立計數）
streak_level 配比：low 12 條、mid 5 條、high 3 條
```

deep_night 重構（33 條）可指定：
```
類別 2 自言自語：15 條（whisper 10 / speech 5）
類別 C 撒嬌：8 條
類別 1 思考：5 條
類別 B 詢問：3 條（binary：「還不睡？」）
類別 E 情境旁白：2 條
```

#### 對 §7 schema 的影響（內容分類元資料）

選擇性加 `_meta.content_class` 欄位（值為 `"1"`-`"4"` 或 `"A"`-`"E"`）便於後續統計與 LLM 補句時對齊風格。**不影響執行邏輯**，純元資料。

```jsonc
{
  "sequenceId": "haiyin_click_004",
  "type": "speech",
  "interaction": "choice",
  "_meta": {
    ...
    "content_class": "3",  // 主動互動 / 閒聊
    "streak_level": "low"
  }
}
```

## §11 風險與緩解

| 風險 | 機率 | 影響 | 緩解 |
|---|---|---|---|
| 時間音串接銜接不自然（停頓、音量差）| 中 | 中 | 統一 ref audio 預生確保音色一致；padding 150ms 加緩衝 |

| Game_enter 跟 game_silence 順序衝突 | 低 | 中 | game_enter priority 80（一般觸發層），game_silence 999（壓制層）。state_edge 觸發那一秒仍會被 suppress 擋。要在 trigger-engine 邏輯裡讓 state_edge 優先於 suppressor 評估，或 game_enter 改 priority 1000（exempt） |
| voice manifest 變大、查詢慢 | 低 | 低 | 預期條目 ≤ 200，O(1) lookup 沒問題 |
| dialogues schema 改動破壞既有 director | 中 | 高 | 加 schema version migration；既有 sequence 沒 choices 欄位仍正常運作 |
| 你補不齊 8 個 category 句 → 部分觸發冷清 | 中 | 中 | placeholder 5 句先頂著；先把架構 ship 上，內容慢慢補 |

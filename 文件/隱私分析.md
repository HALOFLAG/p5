## 一、威脅模型 — 「誰可能看到資料、看到什麼」

要先界定「我們在防誰」，否則保護機制無從設計。

| 威脅來源 | 可能看到 | 風險等級 | 緩解難度 |
|---|---|---|---|
| **使用者本人** | 全部本地資料 | 信任對象 | — |
| **同住家人 / 室友** | 本機 `data/events/*.jsonl`、`config/*` | 中 | 易（檔案加密、權限） |
| **公司 IT / 系統管理員** | 所有本機檔 + 網路流量 | 高 | 難（IT 有 root） |
| **AI provider（Anthropic/OpenAI）** | Stage 1 上傳的事件時序 | 中 | 中（選 zero retention 條款） |
| **本地駭客（病毒、惡意程式）** | 本機檔、API 金鑰 | 中 | 中（防毒、加密、最小權限） |
| **MITM 中間人攻擊** | API 流量 | 低 | 易（HTTPS） |
| **政府傳票 / 司法調閱** | AI provider 那邊的紀錄 | 低（看法域） | 難（純政策） |
| **AI provider 內部員工** | 上傳資料 | 低 | 中（zero retention 限制存取） |
| **未來雇主 / 保險公司** | 不可能直接取得，但若資料外洩理論可被建檔 | 極低 | — |

**對「自用」場景**：主要威脅是「同住人 + AI provider + 本地駭客」這三層。
**對未來「商業化」場景**：威脅範圍擴大到「所有使用者的家人/IT/法院」，每一層都要有政策回應。

## 二、收集了什麼 — 敏感度分級

並不是所有「行為資料」敏感度相等。分級：

### 🟢 低敏感（接近匿名統計）
- 滑鼠點擊**次數**
- 鍵盤按鍵**次數**（不記內容）
- 閒置時段**起訖時間戳**
- 連續使用**時長**

這些單獨看是匿名的；但是**累積夠久**就會變成個人特徵。

### 🟡 中敏感（可識別個人習慣）
- 一天中各時段的活動分佈（→ 推斷作息）
- 滑鼠軌跡 / 打字節奏（→ keystroke dynamics 是學界已知生物特徵，可用於身份驗證）
- 前景應用程式**名稱**（→ 推斷職業、興趣）
- 全螢幕事件起訖（→ 推斷遊戲/影片偏好）

### 🔴 高敏感（直接洩漏隱私）
- **視窗標題**包含的內容：
  - email 主旨：「**信用卡帳單 - 4/12 應付 NT$28,000**」
  - 銀行網頁：「**XX銀行 - 餘額 1,234,567**」
  - 私訊預覽：「**LINE - 媽媽：你爸最近...**」
  - 醫療紀錄：「**XX醫院 - 王小明 病歷編號 12345**」
  - 私密文件：「**離婚協議書.docx**」
  - 公司機密：「**A公司收購 B公司 機密合約.pdf**」
  - 政治/宗教/性傾向相關網頁標題
- 應用程式可洩漏：色情、宗教、政治、約會 app 等高度個人面向

### ⚫ 不收集（明確排除）
- 鍵盤按鍵**內容**（哪個字）
- 剪貼簿
- 螢幕截圖
- 檔案內容
- 網路流量
- 麥克風 / 攝影機

## 三、資料流向 — 三階段詳細

### Stage 0：Logger（完全本地）
```
[uiohook 攔到事件] → [過濾敏感模式 regex] → [寫入 data/events/2026-05-08.jsonl]
        ↑                       ↑
        |                       └─ email/URL/信用卡格式遮罩
        |
        └─ 鍵盤只記時間戳+是否修飾鍵，不記字
```

**這層**：完全離線，零外傳。風險主要是「**檔案被同住人 / IT / 駭客讀到**」。

### Stage 1：批次預生成（**這是最敏感的一步**）
```
[本機讀 30 天 events 檔]
        ↓
[聚合成統計摘要 + 採樣典型事件時序]
        ↓
[組成 prompt：人格描述 + 觀察摘要]
        ↓
[HTTPS POST → Claude/OpenAI API]
        ↓
[AI 回傳 1500 句台詞 JSON]
        ↓
[本機寫入 personas/<x>/dialogues.json]
```

**這層的風險**：
1. 上傳給 AI provider 的內容**就是離開你電腦的全部資料**
2. provider 可能（依條款）保留 30 天作為服務監控
3. provider 內部員工**理論上**可調閱（雖通常不會）
4. 政府傳票可調閱該 30 天保留資料

**關鍵設計決策**：你選的「**詳細事件時序**」上傳粒度，意味著 AI 會看到：
- 「使用者在 4/12 23:47 開了 chrome.exe，視窗標題遮罩後是 [銀行網站]」
- 「使用者每週四晚上 21:00-23:00 是 ××軟體高活躍期」

這比「只上傳統計摘要」的方案多揭露不少資訊。是你**為了個人化深度做的取捨**。

### Stage 2：Runtime（完全本地）
```
[載入 personas/<x>/dialogues.json] → [規則查表] → [顯示氣泡]
```

**零外傳**。執行期完全離線。

## 四、風險場景 — 具體想清楚會發生什麼

### 場景 A：你失戀，那一週每天看分手歌單
- Stage 0：events 檔記下「Spotify - 失戀情歌 24 hours」
- Stage 1：上傳給 AI 時這個視窗標題在內
- 後果：AI provider 那邊有 30 天紀錄；產出的台詞可能會引用「最近常聽憂鬱音樂」 → 情感處理是好事但**資料留在外部** 30 天

### 場景 B：你工作上看了商業機密合約
- Stage 0：events 檔記下「Word - A公司收購B公司機密合約」
- Stage 1：上傳給 AI provider
- 後果：**這違反保密協議**；雖然 AI provider 不會主動洩漏，但「商業機密被傳到第三方雲端」這件事本身就違約

### 場景 C：你電腦被家人借去用
- 角色出現，講「你昨晚 3 點還在追劇」
- 家人看到設定面板的「最近觸發紀錄」「事件統計」
- 後果：作息曝光、私人 app 使用曝光

### 場景 D：你電腦中毒
- 病毒可讀 `data/events/*.jsonl`
- 也可讀 `config/settings.json`（含 AI provider API 金鑰）
- 後果：**API 金鑰外洩 → 駭客拿你的帳戶呼叫 AI** → 帳單暴增

### 場景 E：政府傳票 / 訴訟調閱
- 假設你住的法域有強制調閱權
- Stage 1 在 AI provider 那的 30 天紀錄可被調閱
- 後果：「使用者在 X 天 X 時做了 X」這種事件時序被當作呈堂證供

### 場景 F：AI provider 資料外洩（駭客攻擊 OpenAI）
- 你 30 天前的事件時序在他們資料庫
- 後果：你的活動模式跟著 OpenAI 用戶清單一起進黑市

### 場景 G：未來換工作背景調查（極低機率但存在）
- 雇主買到的「資料聚合商」資料庫**可能**有你的應用使用習慣
- 雖然這條路徑現在還不是主流，但 OSINT 趨勢在走這個方向

## 五、保護機制（技術層 + 政策層）

### 技術層 — 應該做的事

| 機制 | 目的 | v1.1 是否已涵蓋 | 建議 |
|---|---|---|---|
| 視窗標題 regex 過濾（email、URL、信用卡） | 攔最低層的明顯敏感字 | ✅ §11.3 | 加 IBAN、護照號、身分證格式 |
| **應用程式黑名單**（如：不記 keepass、銀行 app、private browser） | 從源頭排除 | ❌ 未提 | **應加進 v1.2** |
| **本地 events 檔加密**（AES-256） | 防同住人/駭客讀 | ❌ 未提 | **應加進 v1.2** |
| **API 金鑰存於 OS keychain** | 防設定檔被讀 | ❌ 未提 | **應加進 v1.2**（用 keytar 或 Windows Credential Manager） |
| 資料保留時限（30 天歸檔、60 天刪） | 最小化暴露窗口 | ✅ §11.2 | 已有 |
| **上傳前預覽 + 逐項拒絕**（使用者看到「我即將傳這些給 AI」） | 知情同意 + 透明 | ✅ §11.3 | 已有，但要實作好 UI |
| **去識別化選項**（事件時序中時間戳被 ±N 分鐘 jitter） | 降低時序攻擊 | ❌ 未提 | 可選加 |
| **本地 LLM 選項** | 完全離線替代 | ✅ §15 | 已抽象 provider，要實作 ollama provider |
| **設定檔備份不含 API 金鑰** | 防 .bak 洩漏 | ❌ 未提 | **應加進 v1.2** |
| HTTPS + 證書釘選 | 防 MITM | 預設 | electron 預設有 |

### 政策層 — 應該寫進 README / 隱私聲明

- 明確列「會收什麼、不會收什麼」
- 明確列「上傳到誰、保留多久」
- 一鍵「**清除所有 events 並重置**」按鈕
- 一鍵「**匯出我的所有資料**」按鈕（GDPR 要求）
- 首次啟動的「**知情同意**」流程（不是惱人彈窗，而是設定流程的一步）

## 六、法律合規 — 不同地區要求

| 法域 | 主要法規 | 對你的影響 |
|---|---|---|
| 台灣 | 個人資料保護法 | 自用無問題；商業化要有隱私聲明、特種個資告知 |
| EU | GDPR | 商業化最嚴：必須有 DPO（資料保護長）、跨境傳輸條款（EU→US 要 SCC）、72 小時資料外洩通報 |
| 美國加州 | CCPA / CPRA | 必須有「不販售我的資料」開關、刪除權、知情權 |
| 中國 | 個保法 PIPL | 出境傳輸要評估（本地 LLM 模式才能合規） |
| 日本 | 個人情報保護法 | 中等嚴格，敏感資料定義較寬 |

**對「自用」**：法律基本不適用（你自己的資料自己處理）。
**對「商業化」**：每一條都要回應，建議：
- 一開始只做台灣 + 日本市場
- 海外市場提供「**完全本地 LLM 模式**」作為合規通路
- EU 市場慎入（除非 ROI 足以攤合規成本）

下次改版要加這些。

- 應用程式黑名單機制（白名單也支援）：使用者可指定某些 app 完全不記

## 九、給你的具體建議

### 對「自用」階段
- 把 v1.2 隱私清單裡的 **API 金鑰存 keychain + events 檔加密 + 預設本地 LLM** 三項落實，其他可延後
- 同住人 / IT 風險：把 `data/` 設成 OS 加密磁區（Windows BitLocker）
- 用 zero-retention 雲端 provider

### 對「未來商業化」階段
- **預設模式必須是「完全離線本地 LLM」**，這是合規護城河
- 雲端 provider 走「明確 opt-in + 詳細告知」流程
- EU 市場慎入；亞洲市場為主
- 必須有專人法務支援，不能 DIY 寫隱私聲明

## 附錄：隱私 = 信任 = 留存率

最後值得一提的是：對「桌寵」這種「安裝在電腦上常駐的東西」，**使用者的潛在隱私焦慮比一般軟體高**。
這個專案要在隱私上做到「**透明 + 可關 + 可刪 + 可離線**」四個極限，不只是合規，更是**使用者願意長期留它在桌面**的前提。

簡言之：
- **透明**：我看得到你看到什麼
- **可關**：我不想分享的我可以不分享
- **可刪**：我反悔了可以全部清掉
- **可離線**：完全不上網也能用

四個都做到，使用者安心；缺一個，桌寵就會被刪。
